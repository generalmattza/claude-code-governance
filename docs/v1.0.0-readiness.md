# v1.0.0 Readiness

The project is feature-complete and infrastructure-complete as of `v0.9.0-rc.2`. Hooks, settings, deployment guides, threat model, ADRs, release engineering, and pilot-validation runbook are all shipped.

`v1.0.0` ships when the maintainer completes the following user-action items. None of these can be performed by Claude Code; they require humans, fleets, money, and credentials.

## Required for v1.0.0

1. **Pilot validation** per `docs/pilot-validation.md`. Six-week engagement with a real BITSUMMIT customer (a regulated municipal agency, a regional law-enforcement service, a federally-regulated research organization, or other). Final report signed and filed.
2. **External security review** per `docs/external-security-review-rfp.md`. Paid engagement with a recognized security firm. Public-release version of the report attached to the `v1.0.0` GitHub release notes.
3. **PGP release key generated.** Fingerprint published in `SECURITY.md`. Public key uploaded to `keys.openpgp.org` and `keybase.io`. CI secret `GPG_PRIVATE_KEY` and `GPG_PASSPHRASE` provisioned in repo settings.
4. **`NPM_TOKEN` provisioned** in repo secrets. First publish to `npmjs.com` confirmed (`npm publish --provenance` was wired in Plan 6; SBOM attaches automatically per Plan 9).
5. **Apple Developer ID cert provisioned.** macOS SEA binary (`ccsec-macos-arm64`, `ccsec-macos-x64`) signed and notarized; Gatekeeper passes. CI secrets `APPLE_CERT_P12`, `APPLE_CERT_PASSWORD`, `APPLE_TEAM_ID`, `APPLE_ID`, `APPLE_APP_SPECIFIC_PASSWORD` provisioned.
6. **`CHANGELOG.md [1.0.0]` entry written.** Includes pilot summary, security review summary, signing fingerprint, and any v1.0.0-only behavioral changes.
7. **Plugin marketplace listing submitted to Anthropic** and approved. Listing references the GitHub repo, the npm package, and the signed binaries.

## Optional but recommended

These do not block `v1.0.0` but are strong signals of project health.

- **Windows EV cert.** Authenticode signature for `ccsec-windows-x64.exe` without SmartScreen warning. CI secrets `WINDOWS_CERT_PFX`, `WINDOWS_CERT_PASSWORD` provisioned.
- **OpenSSF Scorecard.** Configuration verified, badge unblocked from "pending Plan 9" placeholder in the README.
- **Public case study** from the pilot client (with permission). One page, with editorial review by the client's communications team.
- **First v1.x advisory dry-run.** Before the project hits real-world use, the maintainer should do an end-to-end test of the GHSA pipeline with a synthetic advisory to confirm the runbook works.

## Ship procedure

When all required items are complete, the maintainer runs:

```
git tag -a v1.0.0 -m "v1.0.0: pilot-validated, externally-reviewed, signed"
git push origin v1.0.0
gh release create v1.0.0 \
  --repo Bitsummit-Corp/claude-code-security \
  --title "v1.0.0 - Production Ready" \
  --notes-file release-notes/v1.0.0.md
```

The `release.yml` workflow (Plan 9) will then automatically:

- Publish to npm with provenance.
- Generate SBOM via `@cyclonedx/cyclonedx-npm` and attach to the release.
- Build SEA binaries for macOS arm64 / macOS x64 / Linux x64 / Windows x64 via `scripts/build-sea.sh`.
- Generate `SHA256SUMS` across binaries and attach to the release.
- Apply PGP signature to `SHA256SUMS` if `GPG_PRIVATE_KEY` is provisioned.
- Apply Apple Developer ID signature + notarization to macOS binaries if `APPLE_*` secrets are provisioned.
- Apply Authenticode signature to Windows binary if `WINDOWS_*` secrets are provisioned.

If a required signing secret is missing, the workflow degrades gracefully (skip the signing step, attach the unsigned binary, log a warning) rather than fail the release. The maintainer is expected to provision missing secrets and re-run the failed jobs before announcing `v1.0.0` publicly.

## Why this gating exists

The project's value proposition is "trustworthy hardening reference for regulated environments." A `v1.0.0` that has not been deployed in production and has not been third-party reviewed cannot deliver that value, regardless of test count or hook count. The gating is intentional, not bureaucratic.

When the user-action items complete, the gating dissolves and `v1.0.0` is a one-command ship.
